## 智能体封装

### 思路

逻辑如下:

![image-20250622153449013](./assets/image-20250622153449013.png)

先讲其进行归类



之后按照如下思路进行整理

创建一个大模型的core,

里面有工厂类

智能体基类





```
上层接口  ->

业务流: WebOrchestrator()  -> analyze_image_to_scripts() [ 设置运行时, 智能体注册,消息机制发送 ]

大模型的base: gui-tars,qwen
各个智能体: [  图片智能体,yaml智能体,脚本执行智能体,数据库保存智能体  ]
智能体工厂: 将上述智能体注册到运行时中,assistant的创建
运行时:
每个智能体: 创建assistant, 创建team, 对运行时的消息进行处理   -> 继承一个基类
智能体记忆: 也有一个基类,可以记录一个用户,不同次对话时多个智能体的所有记忆
```







每个服务的组成

```
xxx_service:
	- 业务:业务入口,创建运行时,发布消息
	- 各个智能体,继承封装好的基类 (业务处理,存储到数据库,执行脚本)
	- 智能体,从智能体工厂创建Assistant,每个Assistant的model由core中的llm导入,实现自己的操作
	- 运行时的类: 管理消息的发布
```





### 整理

和AI对话内容如下:

```
我想要逐步优化后端的框架,请和我一起一步步优化,先完成第一步,当前完成了AI对话服务,用例生成服务,UI测试服务,当未来完成的服务更多时,在一个services文件夹下,会非常的混乱,请对backend/services 做整理,如backend/services/autogen_service.py服务为AI对话,则在services下创建一个文件夹,将AI对话服务相关的代码放在里面管理,其他服务做相关的操作
```
